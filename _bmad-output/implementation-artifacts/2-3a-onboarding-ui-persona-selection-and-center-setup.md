## Story

As a **newly-verified user landing on the ClassLite dashboard for the first time**,
I want to **pick which persona describes my role and name + brand my center in a single, resumable flow**,
so that **the platform tailors the rest of the wizard to my role, my center has a recognizable identity, and I can walk away mid-flow without losing what I typed**.

## Response Envelope Contract (inherited from Story 2.1)

Every 2xx response from the four consumed endpoints returns `{ data, meta: { serverTime: ISO-8601 UTC } }`. **`apiFetch` drops `meta` on the floor** (`parseEnvelope<T>` at `src/lib/api-fetch.ts:200-228` returns `body.data` only — Winston-W1 verified) — the wizard's "Auto-saving · last saved Ns ago" affordance is driven instead by the `OnboardingProgressResult.updatedAt` field on the PUT/GET progress `data` payload. Local delta is `Date.now() - Date.parse(result.updatedAt)`, refreshed on a lightweight interval. Errors return `{ error: { code, message, requestId, details? } }`. `apiFetch` throws `ApiError` on failure — the wizard's mutation `onError` handlers branch on `ApiError.status + ApiError.code`.

## Acceptance Criteria

1. **Route `/welcome` (screen s00) renders a full-bleed onboarding shell.** New route mounted OUTSIDE `AppLayout` — no sidebar, no topbar; only the wizard-scoped `<OnboardingLayout>` chrome (ClassLite wordmark top-left; `<email> · Sign out` top-right; dot-grid background pattern per §5). The three persona cards render **2-column at `md`, 3-column at `lg`+** (Sally-S5 fold — Vietnamese Founder headline at ≥230px card width overflows; 2-col at `md` gives each card ~350px). Cards stack vertically below `md`. Each card contains: (a) color-keyed SVG relationship illustration (amber for Operator, green for Founder, blue for Solo Teacher — see wireframe `s00`, illustrations `aria-hidden="true"` so SR does not read text nodes "YOU"/"STUDENTS"/"CENTER 3 TEACHERS"), (b) Fraunces+italic title, (c) short lede. **Zero-selection on first paint** (Sally-B1 fold): NO persona has `aria-checked="true"` initially; the focus ring lands on the first card via roving-tabindex per WAI-ARIA APG, but selection is empty until user presses Space/Enter or clicks. If GET `/api/onboarding/progress` returns a non-null `persona`, THAT persona IS the initial `aria-checked` selection (resume-safe). One "Continue →" primary button, **genuinely disabled until a selection exists** (real seatbelt, not defensive-coding belt).

2. **Persona card interaction — selection is single-select radio semantics.** Clicking a card sets it selected + expands the description block below the lede (short paragraph clarifying scope: e.g. Operator = "You orchestrate. They teach." → full description on selection). Deselecting the previously-selected card is automatic. Cards are keyboard-navigable via arrow keys (`role="radiogroup"` + `role="radio"` per WCAG 2.1 AA). `aria-checked` reflects the current selection. Only ONE card is `tabbable` at any time (the selected one); the others receive `tabIndex={-1}` per the ARIA authoring practices radio-group pattern.

3. **Continue from `/welcome` — persist persona + advance step + navigate.** Clicking "Continue →" fires this sequence: (a) `POST /api/onboarding/persona` with `{ persona: "operator" | "founder" | "solo_teacher" }` — idempotent per Story 2.1 AC1, so a re-click on the same value is safe; (b) on success, `PUT /api/onboarding/progress` with `{ currentStep: "center", payload: { schemaVersion: 1, personaChoice: <selected>, centerDraft: null, templateDraft: null } }` to bookmark the resume point; (c) on success, `navigate("/setup/center", { replace: true })` (replace so the back button does NOT re-render `/welcome` with a stale radio state after progress advances). Any error at step (a) or (b): render an inline `Alert` above the button with the localized error message; do NOT auto-retry; do NOT navigate. `401` is caught by the global `apiFetch` refresh path (component never sees it). `429 RATE_LIMIT_EXCEEDED` renders `t('onboarding.persona.error.rateLimited')` with the `Retry-After` seconds interpolated. `403 EMAIL_VERIFICATION_REQUIRED` renders `t('onboarding.persona.error.emailVerificationRequired')` — this is an authorization failure caused by an unverified email, and the user should be routed to `/verify-email?pollId=<their-pollId>` via an in-page link (the pollId isn't recoverable client-side without a server round-trip; if unavailable, the link points to `/verify-email` — Story 1.9a's page handles the missing-pollId case).

4. **Route `/setup/center` (screen s01) renders the center-setup form inside the same onboarding shell.** Layout: single-column `setup-card` (max-width ~560px, centered). Card header: dot step-progress ("Step 2 of 4" for Operator/Founder; "Step 2 of 3" for Solo Teacher — the last dot fades because the Solo flow is shorter per UX §8.1). Card title: Fraunces "Name your center". Card subtitle: descriptive lede. Persistent auto-save affordance at the top-right of the shell reads `Auto-saving · last saved {{n}}s ago` (localized), driven by the last successful PUT progress `result.updatedAt` (see Response Envelope Contract — NOT `meta.serverTime`, which never reaches the caller). **Pristine state** (no save has fired yet in this session): render `t('onboarding.wizard.autoSaving.idle')` = "Auto-save on" (Sally-B2 fold — NOT "Not yet saved," which reads as error on fresh mount). **Failed state**: render `Save failed — retrying…` with an amber icon distinct from success. **After ~5s of consecutive failures**, elevate to a persistent inline banner `t('onboarding.wizard.autoSaving.failedPersistent')` = "Can't reach the server. Your draft is safe in this tab — keep going." Banner clears when a save succeeds.

6. **Auto-save behaviour — debounced PUT of the current draft.** On any tracked field change (center name OR brand color), fire a debounced `PUT /api/onboarding/progress` after **1500ms** of no change with `{ currentStep: "center", payload: { schemaVersion: 1, personaChoice: <resolved>, centerDraft: { name: <trimmed>, brandColor: <selected|null>, logoUrl: null }, templateDraft: null } }`. Auto-save is a **fire-and-forget** mutation — the wizard does NOT block on it. On success, update the shell's `Auto-saving · last saved Ns ago` timestamp from `meta.serverTime`. On failure, render `Save failed — retrying…` in the same slot and try again on next debounce cycle (idempotent PUT). The debounce timer clears on unmount and on route change to prevent stale writes after navigation. **Do NOT auto-save empty forms** (skip PUT if `centerDraft.name` is empty AND user hasn't touched any field — prevents a burst of no-op writes on first paint).

7. **Save & continue — center creation is the transactional step.** Clicking "Save & continue →" submits the form. Sequence: (a) client Zod validation — invalid → RHF field errors, no network call; (b) `POST /api/centers` with `{ name, brandColor, logoUrl: null }` — the `brandColor` value is always non-null (radio has a default); (c) on 201 success: (c.i) write the server-returned `{ id, name, shortCode, brandColor, logoUrl, timezone }` into the auth session cache as a new `Session.center` slot (see AC9), (c.ii) `PUT /api/onboarding/progress` with `{ currentStep: "template", payload: <same as auto-save + centerDraft populated> }` to advance the resume bookmark, (c.iii) `navigate("/setup/template", { replace: true })` — Story 2.3b owns that route; until 2.3b ships, `NotFound` renders (documented as an accepted intermediate state in the change log). Errors:
   - `409 USER_ALREADY_HAS_CENTER` → Sally-S4 fold, two-line recovery: line 1 = `t('onboarding.center.error.userAlreadyHasCenter')` = "Your account already has a center: `{{centerName}}` (`{{shortCode}}.classlite.app`)" (interpolated); line 2 = `t('onboarding.center.error.userAlreadyHasCenter.hint')` = "Not familiar? [Contact support →](/support?type=account-security) — otherwise open your dashboard." Both `{{centerName}}` + `{{shortCode}}` come from the 409 response body — **FU-2-3a-F** tracks verifying Story 2.1's 409 shape carries both; if the fields are absent, patch backend before green-phase. Covers benign race AND compromised-account scenario.
   - `422 VALIDATION_ERROR` with `details: [{ field, message, code? }]` → iterate details, call `form.setError(field, { message })` for known fields (`name`, `brandColor`, `logoUrl`). Unknown-field details → form-level `Alert`. Field-name mapping follows the RegisterPage pattern verbatim.
   - `403 EMAIL_VERIFICATION_REQUIRED` → inline `Alert` with a link to `/verify-email`.
   - `429 RATE_LIMIT_EXCEEDED` → inline `Alert` with the `Retry-After` seconds interpolated + disable the submit button for that many seconds. `apiFetch.ApiError.retryAfterSeconds` (already parsed per Story 1-8) carries the value.
   - `500 INTERNAL_ERROR` → generic `Alert` with `t('onboarding.center.error.generic')` + the `requestId` visible in Geist Mono. **Do NOT auto-retry** — the write may have partially landed (Story 2.1 AC5 pins slug-collision retry to 5 attempts; a 500 means all 5 collided, which is either a hostile actor or a bug worth investigating, not a retry surface).
   - Any error path: keep the form values (do NOT reset), keep the auto-save timer running, allow re-submit.
